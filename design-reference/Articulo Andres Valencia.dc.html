Animaciones que comunican: GSAP con criterio

A
Andrés Valencia
Full Stack Developer
portada · 1600×900

Una animación no es decoración. Es una forma de explicar al usuario qué cambió, hacia dónde mirar y qué acaba de pasar. Cuando se usa con intención, dirige la atención sin que nadie se dé cuenta.

En este artículo comparto cómo decido cuándo animar y cuándo no, y cómo implementarlo con GSAP y ScrollTrigger sin caer en el exceso. La regla que sigo es simple: si una animación no comunica nada, sobra.

1. Cada animación responde una pregunta

Antes de escribir una sola línea me pregunto qué duda del usuario resuelve el movimiento. Un elemento que entra desde abajo dice "esto es nuevo, acaba de aparecer". Un resaltado que se desliza dice "lee esta palabra". Si no hay pregunta, no hay animación.

  • Entrada por scroll: confirma que hay más contenido al bajar.
  • Resaltado de texto: jerarquiza sin gritar.
  • Botón magnético: comunica que el elemento es interactivo.

2. Un reveal honesto con ScrollTrigger

Este es el patrón que uso para revelar bloques al entrar en pantalla. Corto y reutilizable:

import gsap from "gsap";
import { ScrollTrigger } from "gsap/ScrollTrigger";

gsap.registerPlugin(ScrollTrigger);

// Revela cada bloque cuando entra en el viewport
gsap.utils.toArray("[data-reveal]").forEach((el) => {
  gsap.from(el, {
    opacity: 0,
    y: 44,
    duration: 0.85,
    ease: "power3.out",
    scrollTrigger: { trigger: el, start: "top 87%" },
  });
});

Fíjate en el start: "top 87%": la animación arranca cuando el bloque está casi dentro de pantalla, no antes. Así el movimiento acompaña al scroll en lugar de adelantarse.

4. Rendimiento: anima solo lo barato

Animo casi siempre transform y opacity, que la GPU resuelve sin recalcular el layout. Evito animar width, top o height salvo que no haya alternativa. Y respeto siempre prefers-reduced-motion: quien pide menos movimiento, recibe menos movimiento.

Esa es toda la filosofía. Menos animaciones, pero cada una diciendo algo. Criterio antes que cantidad.
