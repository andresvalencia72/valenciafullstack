### PR1 apply findings (2026-07-05)

- **env.ts/env.public.ts tests landed in PR1, not deferred to 2.9.** Vitest's v8 coverage provider (v4.x) has no `all:false`-style opt-out — it always reports every file matched by `coverage.include`, regardless of whether a test touches it. So an untested-but-eligible `env.ts` would have failed the 80% gate immediately in PR1, contradicting the "auto-pass" guarantee. Rather than fight the tool, PR1 ships full unit tests for `env.ts` and `env.public.ts` (fail-fast on missing `DATABASE_URL`/`VISITOR_HASH_SECRET`, `EMAIL_DRIVER` default/enum, build-safe `NEXT_PUBLIC_SITE_URL` default/validation). **Task 2.9 is now redundant** — treat it as "verify/extend existing env.ts tests" rather than "create from scratch" when PR2 lands. The "auto-pass when zero coverage-eligible files exist" behavior itself was verified independently (`vitest run --coverage --coverage.include="<empty-glob>"` exits 0 with an empty 0/0 report), so the gate mechanism in task 1.6 is correct for genuinely-empty scopes — it just doesn't apply once `env.ts` exists.
- **eslint-plugin-boundaries@6.0.2 capture-scoped selector gotcha**: the legacy tuple selector form `["domain", { captured: { feature: "{{from.captured.feature}}" } }]` silently never matches (in either direction) when used inside `allow`/`disallow` — no error, no warning, just a no-op that makes the whole rule permissive-by-default-disallow (i.e., everything gets rejected, or with `default: allow`, nothing gets caught). The working form is the object/`DependencySelector` wrapper: `{ to: { type: "domain", captured: { feature: "{{from.captured.feature}}" } } }`. Verified empirically (see `eslint.config.mjs` inline comment) with a throwaway fixture feature exercising same-feature-allowed / cross-feature-blocked / layer-blocked cases before removing the fixture. Also required `settings["import/resolver"] = { typescript: true, node: {...} }` — without a resolver, boundaries can't resolve `@/*`-aliased or extensionless relative imports and silently treats every dependency as "unknown" (never checked).
- **Folder tree correction applied**: `create-next-app --src-dir` nests `app/` under `src/app`, but design.md's tree places `app/` at the repo root as a sibling to `src/` (screaming architecture: `app/` = composition root only, `src/` = features + shared). PR1 scaffolds with `--src-dir` for the initial `create-next-app` run (so Tailwind/TS/ESLint defaults are generated normally) then moves `src/app` → `app/` immediately after, before writing any feature code. Path alias `@/*` → `./src/*` is unaffected.
- **`.env.example` could not be created** — the execution sandbox denies writes to any `.env*` path, including example/template files with no real secrets. The full recommended content is documented in `README.md` (Environment variables + Getting started) and in `design.md`'s env matrix; a maintainer with local file access should create `.env.example` from that content in a follow-up commit (or relax the sandbox rule for `.env.example` specifically, since `.gitignore` already special-cases it as safe to commit).

### PR2a/PR2b apply findings (2026-07-05)

Supersedes "PR2 apply findings" from the original (now-closed) PR2 draft — same technical content, reorganized across the two branches:

- **307-vs-308 redirect decision (RESOLVED, PR2b)**: verified against the installed `next-intl@4.13.1` middleware source — it calls `NextResponse.redirect(url)` with no explicit status, and Next.js's own `NextResponse.redirect` defaults to **307** when no status is given (confirmed in `node_modules/next/dist/server/web/spec-extension/response.js`). Since the i18n spec requires a **308** (permanent) for the unprefixed-to-`es` redirect, `proxy.ts` wraps next-intl's middleware output and upgrades any 307 to 308 before returning it, preserving all other headers. Locked in by a unit-tested pure function (`shared/i18n/middleware-rules.ts#upgradeRedirectStatus`) plus an e2e test (`e2e/locale-routing.spec.ts`).
- **`middleware.ts` -> `proxy.ts` rename (Next.js 16, PR2b)**: the pinned Next.js version (16.2.10) deprecates the `middleware` file convention in favor of `proxy`. PR2b ships the file as `proxy.ts` from the start.
- **Real hydration bug found and fixed by the Playwright no-flash test (task 2.7, PR2a)**: the original `useTheme()` implementation read `document.documentElement.dataset.theme` (already mutated by the pre-hydration inline script) via a `useState` lazy initializer, producing a genuine React hydration mismatch. Fixed by always starting client state at the SSR-safe default and syncing the real value in a `useEffect` after mount — the same pattern used by `next-themes`.
- **Theme script placement (PR2a)**: `next/script` with `strategy="beforeInteractive"` inside `<body>`, not a raw `<script>` in `<head>` — Next.js's App Router always injects its own resource hints and stylesheet `<link>` at the very top of `<head>`, ahead of any user-authored head content (framework-internal, not configurable). `beforeInteractive` still guarantees the script runs before hydration and before rendered content.
- **Vitest + Node 25 `localStorage` conflict (PR2a)**: Node's built-in global `localStorage` throws `SecurityError` and shadows jsdom's implementation under Vitest. Fixed via `NODE_OPTIONS=--no-webstorage` on the test npm scripts plus a jsdom `environmentOptions.url`.
- **Vitest + `next-intl/navigation` resolution gotcha (PR2b)**: `next-intl`'s react-client build imports the bare specifier `next/navigation`, which Vite's SSR-external resolver can't resolve since `next` ships no package.json "exports" map. Fixed via `ssr.noExternal: ["next-intl"]` + `resolve.alias["next/navigation"]` pointing at the concrete file.
- **Font loader (`next/font/local`) cannot run under Vitest (PR2a)** — excluded `src/shared/ui/fonts/index.ts` from the coverage `include` scope, same basis as the `shared/db` bootstrap-file exclusion category.
- **Fonts (PR2a)**: Clash Display (500/600/700) and General Sans (400/500/600) from Fontshare, committed as `.woff2` with upstream `LICENSE.txt` files.

### PR8 apply findings (2026-07-07)

- **`ArticleSearchRepository`'s `reconcile()` does both halves of "full reconcile" in one call, not separate upsert/prune methods** — matches the search spec's explicit "an upsert-only sync that never prunes does not satisfy this requirement": a single method makes it structurally impossible for `scripts/sync-search.ts` to call upsert without also pruning. Pruning is diffed in JS (fetch existing (slug, locale) keys, delete whatever isn't in the current entry set) rather than a single SQL `(slug, locale) NOT IN (...)` composite-tuple clause — postgres.js/Drizzle tuple-IN parameter binding is fragile and unfamiliar territory in this codebase; the two-query diff is simple to read, simple to test, and (with a handful of articles) not a real performance concern.
- **`search_vector` refresh uses a bound-parameter `::regconfig` cast, not a raw string-interpolated regconfig** — `setweight(to_tsvector(${regconfig}::regconfig, ${title}), 'A') || ...` with Drizzle's `sql` tag automatically parameterizes `${regconfig}`/`${title}`/etc.; Postgres accepts the explicit cast from a bound `text` parameter to `regconfig` (verified empirically against pglite in `drizzle-article-search-repository.test.ts`, including a query containing `to_tsquery`-special characters `& | :` via `websearch_to_tsquery`, per the search spec's "Query with special characters does not error" scenario). No fallback needed — this is the same pglite/real-Postgres parity PR5b's escape hatch verified in principle, now exercised for real by the first actual `to_tsvector`/`websearch_to_tsquery`/`ts_rank` queries in this codebase.

### PR10 apply findings (2026-07-07)

- **`getGithubActivity` checks the token presence itself, before ever calling the client** — design.md's exact wording ("on API error/timeout, or when `GITHUB_TOKEN` is absent, the use-case returns a cached/empty result") reads as a single unified fallback path, so the use-case (not the `app/` composition root) owns the branch: `{ token: undefined }` short-circuits to `{ kind: "unavailable" }` without constructing a network request at all, and any thrown error from `client.fetchActivity` (non-OK status, timeout, malformed payload) collapses to the identical result via a `try/catch`. `ui` (`GithubActivityPanel`) only ever sees one of two shapes and never needs to know *why* data is missing — matches the contact/engagement/search precedent of keeping this branching in the coverage-gated `application` layer, not the excluded `app/` adapter.
- **No GitHub GraphQL / contributions-calendar call** — the REST API has no public endpoint for the contributions calendar (that data is GraphQL-only, `contributionsCollection`, and typically needs broader scopes); rather than adding a second transport (POST-based GraphQL, which also has murkier Next.js Data Cache semantics for non-GET requests) for one field, "recent activity" is derived from the account's recent public events (`GET /users/{username}/events/public`, capped at ~90 days/300 events by GitHub itself) as `recentContributionCount` — a documented, defensible proxy, not the literal contribution-graph number. Recorded here as a conscious scope/interpretation decision, not a silent gap, since the `github-activity` spec only says "contributions and repository data" without pinning an exact metric.
- **Missing-token path takes the *same* code path as an API failure literally, not just "behaves similarly"**: `getGithubActivity` never constructs a request when `token` is falsy, so there is exactly one fallback result shape (`{ kind: "unavailable" }`) for "no token", "GitHub API 4xx/5xx", "network error", "response.json() parse failure", and "AbortController timeout" — verified by 3 dedicated `application` tests (token absent, client resolves, client rejects) plus 2 `infrastructure` tests (non-OK status, timeout) plus 1 schema-mismatch test.
- **`next build` failure discovered and fixed: the home page needed `export const dynamic = "force-dynamic"`.** `GithubActivityAsync` (the `app/` composition root for this section) calls `getEnv()`, which fail-fasts on a missing `DATABASE_URL`/`VISITOR_HASH_SECRET` (design.md's env-validation rule, unchanged since PR1). The home page (`app/[locale]/page.tsx`) previously had no dynamic-rendering trigger and no `DATABASE_URL`-touching code, so `next build`'s static-generation pass tried to fully prerender `/es` and `/en` at build time and crashed on the missing env vars — this is a real, build-breaking conflict with design.md's explicit constraint ("`next build` MUST NOT require `DATABASE_URL` to be set"). Fixed by forcing the whole `/[locale]` route to render dynamically (`export const dynamic = "force-dynamic"`) instead of adding an ad hoc try/catch around `getEnv()` inside the composition root — the dynamic-rendering fix is also what makes task 10.2's residual note ("fetch-level revalidate, not page-level ISR") actually meaningful: a fully static page would freeze the GitHub API response at the last build forever, whereas per-request rendering with the GitHub client's own `next: { revalidate: 3600 }` fetch option gives genuine hourly cache refresh via Next's Data Cache. Partial Prerendering (which would let only the `Suspense` boundary opt into per-request rendering while the rest of the page stays static) is not enabled in `next.config.ts`, so this is a whole-route tradeoff, not a per-section one — flagged here as a deliberate, build-verified architecture decision, not a silent deviation from the folder tree in design.md (which does not specify home page rendering mode either way).
- **`role="status"` collision caught by the full e2e run, not a unit test**: the first draft of `GithubActivityPanel`'s fallback paragraph and `GithubActivityLoading`'s loading paragraph used `role="status"`, matching a pattern that reads well in isolation. `e2e/contact.spec.ts`'s pre-existing "submits successfully" test uses `page.getByRole("status")` (singular, matches `ContactForm`'s own success/error message) and started failing with a Playwright strict-mode "resolved to 2 elements" error once both the contact form's status message and the github-activity fallback were simultaneously present on the same rendered page (CI/local both run with no `GITHUB_TOKEN`, so the fallback panel is always in the DOM). ## PR12 resolve-blockers: CRITICAL-2 (Lighthouse Performance) fix (2026-07-08)

`verify-report.md`'s terminal re-verify (PR1-PR12 scope) found CRITICAL-2: `quality-pipeline`'s Lighthouse Performance Budget MUST-gate (`npm run lighthouse`, `>= 90`) failing deterministically on `/es` and `/en` (median 88-89, vs. the pre-PR12 baseline's passing 89-90/90), root-caused by the report (tentatively) to PR12's vendored icon components shipping in the home page's client JS bundle via the `Reveal`/`Tilt` (`"use client"`) wrapper around `SkillsSection`.

**Root cause independently re-derived this session — the report's specific bundle-weight mechanism was investigated and disproved, then the real mechanism was isolated empirically before any fix was written:**

1. **Bundle-inclusion theory disproved first.** Built production (`npm run build`) and grepped `.next/static` for icon-specific content (`data-icon`, `SKILL_ICONS`, gradient ids, a distinctive path-data substring) — zero matches. The vendored icon components are plain Server Components (no `"use client"`); `SkillBadge`/`SkillsSection` are also Server Components rendering `Reveal`/`Tilt`'s children in the standard "Server Component passed as children to a Client Component" RSC pattern — their code never shipped in client JS, contrary to the report's stated mechanism.
2. **Hydration-cost-of-the-Reveal/Tilt-subtree theory tested next, also disproved.** Temporarily stubbed `Reveal`/`Tilt` to plain pass-through in `skills-section.tsx` (no client boundary at all around the skill cards), rebuilt, ran a real `npm run lighthouse` against `/es` — score was unchanged (0.88/0.89/0.89, identical to the unmodified branch within noise). Ruled out React hydration/reconciliation cost of the larger DOM subtree as the mechanism.
3. **Controlled A/B against the pre-PR12 baseline, in this same session/sandbox** (`feat/pr11-hardening`, rebuilt and measured fresh, not just cited from the prior report): passed cleanly at exactly 0.90/0.90/0.90 on `/es`, LCP ~3616ms, Render Delay ~3162ms (87%). 4. **Raw markup weight isolated as the actual mechanism.** Temporarily stripped only the rendered `<Icon />` output from `SkillBadge` (kept `Reveal`/`Tilt` and everything else real/unmodified) — score recovered to 0.90/0.90/0.89 (2 of 3 passing, one borderline), confirming the ~29KB of hand-drawn, multi-path/gradient inline SVG markup (`postgresql-icon.tsx` 9.3KB and `php-icon.tsx` 5.6KB are the largest single contributors) landing synchronously in every request's initial SSR HTML is what delays the browser's ability to reach the LCP paint frame under Lighthouse's throttled-CPU mobile preset — not client JS bundle size, not hydration cost of a larger DOM subtree.

**Non-obvious findings (all verified empirically against the real built image and a real local Postgres, not assumed)**:

1. **Turbopack bundles `drizzle-orm`/`postgres`/`gray-matter`/etc. directly into `.next/standalone`'s server chunk files** — they are entirely absent from the traced `node_modules` (confirmed: `grep`ing `.next/server/app/api/contact/route.js.nft.json`'s file list for `drizzle-orm` finds nothing), yet the running app works correctly against a live Postgres — verified end-to-end: `GET /api/engagement/clean-architecture-nextjs` against the built Docker image, network-joined to the local dev Postgres container, returned real aggregate counts (`{"views":8,"reactions":{...}}`), and full security headers (CSP, HSTS, etc.) were present and byte-identical to the non-Docker dev server's output.
2. **`drizzle-kit`/`tsx` are separate CLI processes, entirely outside that Turbopack bundling** — invoking `npm run db:migrate`/`db:sync-search` inside the pure-standalone image failed 3 times in sequence with increasingly specific missing-module errors (`drizzle-orm` → a DB driver → `gray-matter`/`next-intl`/`zod`, the last discovered because `scripts/sync-search.ts` imports directly from `src/` via `tsx`, unbundled, and needs `src/` itself copied into the image too). Root-caused by statically walking the real import graph reachable from `scripts/sync-search.ts` and `drizzle.config.ts` (a small Python AST-free regex walk over `import ... from` specifiers, not guesswork) rather than iterating build-fail-fix blindly past that point — the resulting exact package set (`drizzle-kit`, `tsx`, `drizzle-orm`, `postgres`, `gray-matter`, `next-intl`, `zod`) is installed in an isolated `tools` build stage, pinned to `package-lock.json`'s exact resolved versions.
3. **`npm prune --omit=dev` cannot shrink a Next.js production image** — first attempted approach (copy the full `deps` node_modules into `runner`, then `npm prune --omit=dev` + `npm install drizzle-kit tsx` on top) produced a 2.64GB image; investigation found `next`'s own `@next/swc-*` native compiler binaries (~400MB) are legitimately a *production* dependency (needed for `next build`), so devDependency pruning cannot remove them — `.next/standalone`'s own output tracer, by contrast, already correctly excludes them (no compiler needed to run `node server.js`). Fixed by abandoning the prune approach entirely: ship `.next/standalone` as-is, merge in only the isolated `tools` stage's packages.
4. **`npm install <pkg>` without `--omit=peer` re-pulls a full `next` package via a transitive peer dependency** — adding `next-intl` to the isolated `tools` stage (needed for finding #2 above) silently reinstalled a fresh, full copy of `next` (+ all `@next/swc-*` binaries again) because `next-intl` declares `next` as a peer dependency and npm 7+ auto-installs peers by default; image jumped 636MB → 1.57GB from this one addition. Fixed with `--omit=peer` (verified safe: `scripts/sync-search.ts` only reaches `next-intl/routing`'s `defineRouting()`, never a real call into the `next` package itself). Final image: **758MB**.
5. **`.next/standalone` auto-copies any `.env` file present in the build context at `next build` time** — confirmed empirically with a local (non-Docker) `npm run build`: the repo's real local-dev `.env` (containing a real `DATABASE_URL`/`VISITOR_HASH_SECRET`) appeared verbatim inside `.next/standalone/.env` afterward. This makes `.dockerignore`'s `.env`/`.env.*` exclusion a **hard security requirement**, not just build hygiene — without it, a developer's local secrets would be baked into a shipped, pushed image layer. `.dockerignore` already excludes them; this was verified as the reason why, not assumed.
6. **A `docker compose config` credential-materialization footgun, caught mid-session**: running `docker compose -f docker-compose.prod.yml config` (no flags) interpolates every `env_file:`-sourced value into the printed YAML — this leaked the operator's real local-dev `GITHUB_TOKEN` (a live PAT) and `VISITOR_HASH_SECRET` into this session's tool output/transcript. **Action item for the user, disclosed immediately in-session and repeated here: rotate that GitHub PAT.** Fixed the verification method going forward: use `docker compose config --quiet` (validates syntax/schema, exits 0/1, prints nothing) instead of the bare `config` command whenever real secrets might be in scope.
